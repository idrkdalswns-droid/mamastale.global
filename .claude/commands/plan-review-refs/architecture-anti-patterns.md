# 플랜에서 흔히 빠지는 아키텍처 요소

## 패턴 1: 기존 코드 영향 범위 미분석
```
plan.md: "채팅 API의 응답 형식을 변경한다"
빠진 것:
  - 이 API를 호출하는 클라이언트 코드가 몇 곳인가?
  - 응답 형식 변경 시 깨지는 곳 목록을 작성했는가?
  - 하위 호환성을 유지할 건가, breaking change인가?
  - breaking change라면 프론트와 동시에 배포해야 하는가?
질문: "변경 영향 범위(blast radius)를 파일 단위로 명시했는가?"
```

## 패턴 2: 배포 순서 미고려
```
plan.md: "새 환경변수 3개를 추가하고 새 API를 만든다"
빠진 것:
  - 코드 배포 전에 환경변수를 설정해야 하는가, 후에 해도 되는가?
  - DB 마이그레이션 → 코드 배포 → 환경변수 순서가 맞는가?
  - 배포 중 구 코드와 신 코드가 동시에 돌 때 (롤링 배포) 문제 없는가?
  - 실패 시 롤백 순서는? (코드 → 마이그레이션 → 환경변수 역순?)
질문: "배포 체크리스트를 순서대로 작성했는가? 각 단계 실패 시 롤백 계획이 있는가?"
```

## 패턴 3: 테스트 전략 부재
```
plan.md: "새 기능을 구현한다" (테스트 언급 없음)
빠진 것:
  - 핵심 비즈니스 로직의 단위 테스트? (최소한 이것)
  - 기존 기능이 깨지지 않는다는 보장? (회귀 테스트)
  - 어떤 조건에서 "이 기능이 정상 작동한다"고 판단하는가? (수락 기준)
질문: "이 기능의 '완료 조건'이 명확한가? 무엇이 작동하면 완성이고, 무엇이 깨지면 실패인가?"
```

## 패턴 4: "나중에 리팩토링하자" 패턴
```
plan.md: "일단 page.tsx에 로직을 넣고 나중에 분리한다"
위험:
  - "나중에"는 오지 않는다. 기술 부채로 영구 정착.
  - page.tsx가 이미 700줄인데 여기에 더 넣으면 1000줄+
  - 다음 기능 개발 시 이 코드를 건드려야 하면 → 충돌·버그
질문: "이 계획에 '일단', '나중에', '임시로'라는 단어가 있는가? 있다면 정말 나중에 할 건가?"
```

## 패턴 5: 확장 시나리오 미고려
```
plan.md: "사용자별 설정을 localStorage에 저장한다"
빠진 것:
  - 다른 디바이스에서 접속하면? (동기화 안 됨)
  - 브라우저 캐시 삭제하면? (데이터 손실)
  - 사용자가 1000명이 되면? (서버 저장으로 전환 필요?)
  - 설정 항목이 10개에서 50개로 늘어나면?
질문: "이 설계가 사용자 10배, 데이터 10배에서도 유지되는가? 아니면 나중에 다시 만들어야 하는가?"
```

## 구조 적합성 체크 (모든 plan에 적용)

```
이 plan의 변경이 기존 프로젝트 구조에 자연스러운가:

□ 새 파일의 위치가 기존 디렉토리 구조를 따르는가?
□ 새 함수/컴포넌트의 네이밍이 기존 컨벤션을 따르는가?
□ 새 타입 정의가 기존 types/ 구조와 일관되는가?
□ 새 API 라우트가 기존 라우트 패턴을 따르는가?
□ 관심사 분리가 적절한가? (UI에 비즈니스 로직, API에 렌더링 등 혼재 없는가)

하나라도 기존 패턴에서 벗어나면:
  → 의도적이면 OK (근거 명시)
  → 무의식적이면 🟡 MEDIUM (기존 패턴으로 수정 제안)
```
