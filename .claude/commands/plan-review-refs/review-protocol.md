# 플랜 검수 프로토콜

## 이 시스템이 검수하는 것

```
입력물:
  - plan.md (기능 개발 계획서)
  - 또는 사용자가 구두로 설명한 구현 계획
  - 또는 전략 문서, 기획서, 설계안

입력물이 아닌 것:
  - 기존 코드 (코드 리뷰는 별도 도구)
  - 실행 중인 서비스 (QA 테스트는 별도)
```

## 각 에이전트의 검토 질문 패턴

모든 에이전트는 plan.md를 읽고 다음 구조로 검토한다:

```
1. 명시된 것 중 틀린 것: "이 계획에서 A를 하겠다고 했는데, A는 실제로 작동하지 않는다"
2. 빠진 것: "이 계획에서 B를 전혀 고려하지 않았는데, B 없이는 실패한다"
3. 모호한 것: "이 계획에서 C를 하겠다고 했는데, 구체적으로 어떻게 할지 불명확하다"
4. 더 좋은 방법: "이 계획에서 D 방식을 쓰겠다고 했는데, E 방식이 더 낫다"
5. 의존성 충돌: "이 계획의 F가 기존 G와 충돌한다"
```

## 심각도 기준 (코드가 아닌 계획 기준)

**🔴 치명적 누락 (CRITICAL)**
"이 계획대로 빌딩하면 **반드시** 프론트/백엔드에서 문제가 터진다"
- 계획에 인증 방식이 없는데 보호 라우트를 만들겠다고 함
- DB 스키마 변경이 필요한데 마이그레이션 계획이 없음
- 기존 기능을 깨뜨리는 변경인데 회귀 테스트 계획이 없음

**🟠 주요 허점 (HIGH)**
"빌딩은 가능하지만 **높은 확률로** 재작업이 필요하다"
- 에러 케이스 처리 계획이 없음
- 모바일 대응 계획이 없음
- 성능 영향을 고려하지 않음
- 엣지 케이스가 명시되지 않음

**🟡 개선 제안 (MEDIUM)**
"더 좋은 계획으로 만들 수 있다"
- 더 나은 구현 방식 제안
- 사용자 경험 개선 아이디어
- 비용 절감 방안

**💡 아이디어 (INFO)**
"고려하면 좋을 추가 사항"
- 향후 확장성
- 부가 기능 아이디어

## 누적 전달 형식

각 에이전트 완료 시 다음 형태로 정리:

```markdown
### Pass N 발견 사항

🔴 CRITICAL:
- [PN-C01] {제목}: {상세 설명}

🟠 HIGH:
- [PN-H01] {제목}: {상세 설명}
- [PN-H02] {제목}: {상세 설명}

🟡 MEDIUM:
- [PN-M01] {제목}: {상세 설명}

💡 INFO:
- [PN-I01] {제목}: {상세 설명}

교차 발견:
- P{M}-{id} 관련: {추가 분석}
```

## 최종 보고서 구조

7차 최종 판관이 작성:

```markdown
# 플랜 검수 최종 보고서

## 판정: [🟢 APPROVE / 🟡 REVISE / 🔴 REJECT]
{판정 근거 1문장}

## 반드시 수정할 것 (CRITICAL + HIGH)
1. {이슈} → {수정 방향}
2. {이슈} → {수정 방향}

## 권장 수정 (MEDIUM)
1. {제안}
2. {제안}

## 수정된 plan.md (제안)
{7번의 검토를 반영한 개선된 계획}
```
