# 플랜에서 흔히 빠지는 보안 요소

## "이거 빠지면 빌딩 후 반드시 터진다" 패턴

### 패턴 1: 인증 범위 모호
```
plan.md: "사용자만 볼 수 있는 페이지를 만든다"
빠진 것: 
  - "사용자"가 로그인 사용자? 게스트? 관리자?
  - API 레벨에서 인증을 어떻게 체크하는가?
  - 미들웨어로 일괄 적용? 각 라우트에서 개별 체크?
질문: "이 기능의 모든 API 엔드포인트에 대해, 누가 접근 가능한지 명시되어 있는가?"
```

### 패턴 2: "서버에서 처리"라고만 적음
```
plan.md: "서버에서 검증 후 저장한다"
빠진 것:
  - 무엇을 검증하는가? (타입? 길이? 형식? 비속어?)
  - 검증 실패 시 어떤 에러를 반환하는가?
  - 클라이언트에서도 사전 검증하는가? (UX를 위해)
질문: "입력 검증의 구체적 규칙과 실패 시 동작이 명시되어 있는가?"
```

### 패턴 3: 데이터 접근 범위 미정의
```
plan.md: "사용자의 동화 목록을 보여준다"
빠진 것:
  - 다른 사용자의 동화는 절대 못 보는가?
  - 공개 설정된 동화는? 비공개는?
  - RLS 정책이 이 새 쿼리를 커버하는가?
질문: "사용자 A가 사용자 B의 데이터를 볼 수 있는 시나리오가 발생하지 않는가?"
```

### 패턴 4: 결제/포인트 관련 원자성 누락
```
plan.md: "티켓을 차감하고 동화를 생성한다"
빠진 것:
  - 차감 후 생성 실패하면? 티켓이 사라지나?
  - 동시에 두 번 요청하면? 이중 차감되나?
  - 네트워크 끊김으로 응답 못 받으면? 재시도 시 또 차감?
질문: "모든 결제/포인트 트랜잭션의 원자성과 멱등성이 명시되어 있는가?"
```

### 패턴 5: 위기감지 영향 미고려 (mamastale 특수)
```
plan.md: "새 채팅 기능을 추가한다"
빠진 것:
  - 이 기능에서 사용자가 위기 표현을 할 가능성은?
  - 기존 CSSRS 위기감지 시스템이 이 새 경로에서도 작동하는가?
  - 위기감지가 우회될 수 있는 경로가 생기지 않는가?
질문: "새 기능이 기존 위기감지 시스템의 보호 범위 안에 있는가?"
```
