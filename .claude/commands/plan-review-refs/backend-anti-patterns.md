# 플랜에서 흔히 빠지는 백엔드 요소

## 패턴 1: 에러 시나리오 전무
```
plan.md: "AI에게 동화를 생성하게 한다"
빠진 것:
  - AI API 타임아웃(30초) 시 사용자에게 뭘 보여주나?
  - AI가 부적절한 내용을 생성하면? 필터링 계획?
  - Rate Limit에 걸리면? 재시도? 큐잉?
  - 응답이 예상 형식이 아니면? (파싱 실패)
질문: "이 API 호출이 실패하는 5가지 시나리오와 각각의 대응을 명시했는가?"
```

## 패턴 2: DB 스키마 변경 영향 미분석
```
plan.md: "stories 테이블에 tags 컬럼을 추가한다"
빠진 것:
  - 기존 행들의 tags는? NULL? 빈 배열? 기본값?
  - 이 컬럼을 사용하는 쿼리에 인덱스가 필요한가?
  - 기존 API 응답에 tags가 추가되면 프론트가 깨지지 않는가?
  - 마이그레이션 SQL을 명시했는가?
질문: "DB 변경의 마이그레이션 SQL과 기존 데이터 처리 방안이 있는가?"
```

## 패턴 3: 성능 영향 미고려
```
plan.md: "모든 동화에 좋아요 수를 표시한다"
빠진 것:
  - 동화 목록 조회 시 좋아요 수를 어떻게 가져오나? (N+1?)
  - 실시간 카운트? 캐싱? 비동기 집계?
  - 좋아요가 1만 개인 동화와 0개인 동화의 조회 성능 차이?
질문: "새 기능이 기존 API의 응답 시간을 얼마나 늘리는가?"
```

## 패턴 4: 상태 전이 미정의
```
plan.md: "동화의 상태를 draft → published → archived로 관리한다"
빠진 것:
  - published → draft로 되돌릴 수 있나?
  - archived에서 복구 가능한가?
  - 각 상태에서 허용되는 행동은? (수정? 삭제? 공유?)
  - 잘못된 상태 전이 요청 시 에러 처리?
질문: "상태 전이 다이어그램이 명확하고, 불가능한 전이가 차단되는가?"
```

## 패턴 5: 비용 영향 미산출
