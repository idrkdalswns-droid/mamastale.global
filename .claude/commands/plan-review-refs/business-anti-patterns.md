# 플랜에서 흔히 빠지는 비즈니스 요소

## 패턴 1: 기회비용 무시
```
plan.md: "커뮤니티에 댓글 알림 기능을 추가한다"
질문:
  - 이걸 만드는 3일 동안 안 만들어지는 것은?
  - 결제 퍼널 개선 vs 댓글 알림, 어느 쪽이 매출에 더 기여?
  - 사용자 100명 중 몇 명이 이 기능을 요청했는가? (감이 아닌 데이터)
  - "있으면 좋은" vs "없으면 죽는"의 구분은?
판단: "같은 시간으로 매출에 더 직접적으로 기여하는 작업이 있는가?"
```

## 패턴 2: 매출 영향 추정 없음
```
plan.md: "동화 공유 기능을 개선한다"
빠진 것:
  - 공유 → 신규 유입 → 가입 → 유료 전환의 퍼널을 수치로 추정했는가?
  - 공유 1건당 예상 신규 유입: _명 (근거?)
  - 월 공유 예상 횟수: _회 (근거?)
  - 이 기능으로 인한 월 추가 매출: ₩_ (계산식?)
질문: "이 기능의 매출 영향을 숫자로 추정할 수 있는가? 감이면 감이라고 말하라."
```

## 패턴 3: 무료 사용자에게만 유리한 기능
```
plan.md: "게스트도 동화 3편까지 무료로 만들 수 있게 한다"
질문:
  - 무료 범위를 늘리면 유료 전환 동기가 줄어들지 않는가?
  - 무료 1편 → 3편으로 늘리면 "1편으로 충분" → "3편으로 충분"이 되지 않는가?
  - 무료 확대가 가입 전환율을 올리는 근거는?
판단: "무료 확대가 전환율을 올리는 것인지, 아니면 공짜 사용만 늘리는 것인지?"
```

## 패턴 4: 리텐션 동기 미설계
```
plan.md: "[새 기능]을 추가한다"
질문:
  - 이 기능이 사용자의 재방문 이유가 되는가?
  - 한 번 쓰고 끝인 기능인가, 반복 사용하는 기능인가?
  - 이 기능이 사용자의 "습관 루프"를 형성하는가?
    (트리거 → 행동 → 보상 → 투자)
판단: "이 기능은 일회성인가 반복성인가? 반복성이라면 어떤 보상이 재방문을 유도하는가?"
```

## 패턴 5: "지금 해야 하는가" 질문 회피
```
모든 plan.md에 반드시 적용:

P0: 이것 없으면 론칭 불가 또는 매출 직접 차단 → 즉시
P1: 이것 없으면 핵심 가치 전달 약화 → 이번 스프린트
P2: 있으면 좋지만 없어도 론칭 가능 → 다음 스프린트
P3: 나중에 해도 충분 → 백로그

"이 plan은 P{?}이다"를 반드시 명시하라.
론칭 전이면 P0~P1만 통과, P2~P3은 REJECT.
```
