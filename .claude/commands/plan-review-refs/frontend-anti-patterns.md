# 플랜에서 흔히 빠지는 프론트엔드 요소

## 패턴 1: 로딩·에러·빈 상태 누락
```
plan.md: "동화 목록 페이지를 만든다"
빠진 것:
  - 데이터 로딩 중: 스켈레톤? 스피너? 아무것도 없이 빈 화면?
  - API 에러 시: "다시 시도" 버튼? 에러 메시지 문구?
  - 데이터 0건일 때: "아직 동화가 없어요. 첫 동화를 만들어보세요!"
  - 네트워크 끊김 시: 오프라인 안내?
질문: "이 화면의 4가지 상태(로딩·성공·에러·빈)를 모두 계획했는가?"
```

## 패턴 2: 모바일 대응 누락
```
plan.md: "새 대시보드를 만든다"
빠진 것:
  - 320px(iPhone SE)에서 레이아웃이 깨지지 않는가?
  - 테이블이 있다면 모바일에서 어떻게 표시? 가로 스크롤? 카드 변환?
  - 모달이 있다면 모바일 전체 화면으로 전환되는가?
  - 터치 타겟 44×44px 확보?
질문: "모바일 레이아웃을 별도로 설계했는가, 아니면 '반응형이니까 알아서 되겠지'인가?"
```

## 패턴 3: Server/Client Component 구분 누락
```
plan.md: "사용자 프로필 페이지를 만든다"
빠진 것:
  - 어떤 부분이 Server Component(정적 데이터, SEO)인가?
  - 어떤 부분이 Client Component(인터랙션, 상태)인가?
  - 'use client'를 페이지 전체에 걸 건가, 필요한 컴포넌트만 걸 건가?
  - Client Component가 서버 전용 데이터를 import하지 않는가?
질문: "Server/Client 경계가 명확한가? 'use client'를 어디에 쓸 건가?"
```

## 패턴 4: 기존 상태 관리와의 충돌
```
plan.md: "새 설정 기능을 추가한다"
빠진 것:
  - 이 상태를 어디에 저장? useState? Zustand? URL? localStorage?
  - 기존 useSettingsStore와 겹치지 않는가?
  - 페이지 새로고침 후에도 유지되어야 하는가?
  - 다른 탭에서 변경하면 동기화되어야 하는가?
질문: "새 상태의 생명주기(생성·변경·소멸·동기화)를 명시했는가?"
```

## 패턴 5: 접근성 계획 전무
```
plan.md: "투표/설문 기능을 추가한다"
빠진 것:
  - 키보드만으로 투표 가능한가? (Tab → Enter)
  - 스크린 리더가 "3번 선택지, 현재 45% 득표" 를 읽을 수 있는가?
  - 색상으로만 구분되는 UI가 없는가? (색약 사용자)
  - 버튼인데 <div onClick>으로 만들 계획은 아닌가?
질문: "이 기능을 키보드와 스크린 리더만으로 사용할 수 있는가?"
```
