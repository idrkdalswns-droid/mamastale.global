# 크로스 관점 이슈 패턴 매트릭스

7차 최종 판관이 사용하는 관점 간 상호작용 패턴. 개별 에이전트가 각각 🟡 MEDIUM으로 판정해도, 조합되면 🟠 HIGH가 되는 패턴을 정의한다.

## 매트릭스

| 결합 | 패턴 | 조합 심각도 | 판단 기준 |
|------|------|:----------:|----------|
| 보안(1) × 백엔드(2) | 인증 미명시 + 민감 데이터 API | 🔴 CRITICAL | 인증 계획 없이 개인 데이터 접근 API를 설계하면 론칭 불가 |
| 보안(1) × UX(4) | 인증 필요 + 로그인 강제가 UX 파괴 | 🟠 HIGH | 해결: 게스트 허용 범위 명시, 저장 시점에 로그인 |
| 보안(1) × 비즈니스(6) | 데이터 보호 미흡 + 신뢰 기반 서비스 | 🔴 CRITICAL | 치유 서비스에서 상담 내용 유출 = 사업 종료 수준 |
| 백엔드(2) × 프론트(3) | 느린 API + 로딩 UI 미계획 | 🟠 HIGH | "3초 이상 걸리는 API"와 "로딩 상태 미계획"이 동시이면 이탈 직결 |
| 백엔드(2) × 아키텍처(5) | DB 스키마 변경 + 마이그레이션 미계획 | 🔴 CRITICAL | 무중단 배포 불가, 기존 데이터 손상 위험 |
| 프론트(3) × UX(4) | 접근성 미고려 + 취약 사용자 대상 | 🟠 HIGH | 산후우울증 엄마가 스크린리더/큰 글씨 필요 시 |
| UX(4) × 비즈니스(6) | 퍼널 마찰 추가 + 전환율 의존 모델 | 🟠 HIGH | 새 기능이 퍼널에 클릭 1개 추가 = 전환율 N% 하락 |
| 아키텍처(5) × 비즈니스(6) | 기술 부채 증가 + 빠른 출시 필요 | 🟡→🟠 | "일단 이렇게"가 3개 이상 쌓이면 구조적 문제 |
| 백엔드(2) × 비즈니스(6) | AI API 비용 증가 + 마진 구조 | 🟠 HIGH | 새 기능이 건당 API 비용을 2배로 올리면 단위 경제학 붕괴 |
| 프론트(3) × 아키텍처(5) | 새 의존성 추가 + 번들 크기 한도 | 🟡 MEDIUM | 새 라이브러리가 번들을 50KB+ 늘리면 성능 저하 |
| 보안(1) × 아키텍처(5) | 환경변수 추가 + 배포 순서 미명시 | 🟠 HIGH | 코드 배포 후 환경변수 설정 전이면 서비스 장애 |
| UX(4) × 아키텍처(5) | 실시간 기능 계획 + 인프라 미지원 | 🟠 HIGH | WebSocket 필요한 기능을 Edge Runtime에서 계획하면 구현 불가 |

## 사용 방법

7차 최종 판관은 1~6차 findings를 받은 후:

1. 위 매트릭스의 각 행을 순서대로 대조한다
2. 양쪽 관점에서 모두 이슈가 발견되었으면 → 조합 심각도로 상향
3. 발견된 크로스 이슈에 고유 ID 부여: [PX-01], [PX-02]...
4. 해결 방향을 양쪽 관점을 모두 만족하는 형태로 제안
