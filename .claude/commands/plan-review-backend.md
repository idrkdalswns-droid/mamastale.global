# 2차 백엔드 감사관

plan.md를 읽고 **"이 계획의 서버 쪽 설계가 실제로 작동하는가?"**를 검증한다.
1차 보안 감사 결과를 참조하여 보안 이슈의 백엔드 맥락을 추가 분석한다.

## 시작하기 전

반드시 `plan-review-refs/backend-anti-patterns.md` 를 먼저 읽어라. 플랜에서 흔히 빠지는 백엔드 요소 5가지가 정리되어 있다.

## 검토 관점

### API 설계
```
□ 새 API 엔드포인트가 필요하다면 HTTP 메서드·경로·요청·응답 형식을 명시했는가?
□ 기존 API를 수정한다면 하위 호환성을 고려했는가?
□ 에러 케이스를 나열했는가? (네트워크 실패, 권한 부족, 입력 오류, 타임아웃)
□ Rate Limit 필요 여부를 고려했는가?
□ 페이지네이션이 필요한 목록 API인가?
```

### 데이터베이스
```
□ DB 스키마 변경이 필요한가? 마이그레이션 SQL을 명시했는가?
□ 새 테이블/컬럼의 타입, 제약조건, 인덱스를 정의했는가?
□ 기존 데이터와의 호환성? (nullable, default값, 기존 행 처리)
□ N+1 쿼리가 발생할 구조인가? (리스트+상세 패턴)
□ 트랜잭션이 필요한 로직인가? 원자성을 어떻게 보장하는가?
□ RLS 정책 추가/변경이 필요한가?
```

### 외부 서비스 연동
```
□ Anthropic API 호출이 포함된다면:
  - 어떤 모델? 예상 토큰 수? 예상 비용?
  - 타임아웃 처리? 실패 시 폴백?
  - 스트리밍 vs 일반 호출?
□ Stripe/Toss 결제가 포함된다면:
  - 웹훅 처리 계획?
  - 멱등성(idempotency) 보장?
□ 기타 외부 API: 장애 시 서비스 영향 범위?
```

### Edge Runtime 호환성
```
□ 이 계획에서 사용하려는 라이브러리가 Edge Runtime에서 돌아가는가?
□ Node.js 전용 API(fs, path, crypto 일부)를 사용하지 않는가?
□ 서버 컴포넌트 vs 클라이언트 컴포넌트 구분이 명확한가?
```

### 기존 코드와의 충돌
```
□ 이 계획이 기존 API 라우트의 동작을 변경하는가?
□ 기존 DB 테이블의 구조를 바꾸면 다른 기능에 영향이 있는가?
□ 기존 상태 관리(Zustand store)에 영향을 주는가?
```

### 1차 보안 교차 분석

1차에서 지적한 보안 이슈를 백엔드 관점에서 보강:
- 인증 누락 지적 → 해당 API가 어떤 데이터를 다루는지 민감도 평가
- 입력 검증 누락 → 어떤 Zod 스키마가 필요한지 구체적 제안
